# Spec 010 — Самолечение (триггер ошибки → агент-чинильщик → гейт → одобрение)

> Статус: APPROVED Игорем (шаг 5 roadmap). Ветка `feature-self-healing`.
> ⭐ Вершина roadmap. Рамка: ИНСТРУМЕНТ Игоря + OSS. БЕЗ привязки к Coolify/Vault.
> Строится НА проверенном фундаменте: сканер инцидентов + C2-worktree + гейт качества + Timeline.

## Идея
Соединить уже готовые кирпичи в петлю:
```
сканер ловит падение (лог/тест) → создаёт err-карточку (УЖЕ работает)
   → [НОВОЕ] авто-запуск агента-чинильщика в worktree (C2-изоляция)
   → агент чинит → авто-коммит в ветке
   → [НОВОЕ] гейт прогоняет тесты (Spec 009)
   → safe? → карточка в Review с пометкой «🔧 авто-починка, гейт ✓» + пинг Игорю
   → Игорь жмёт ✓ Применить / ✗ Отмена (C2-гейт)
```
Это НЕ полная автономия — это **«агент готовит фикс, человек одобряет»**. Безопасно: всё в worktree, ничего не applied без Игоря.

## ⚠️ ПРЕДОХРАНИТЕЛИ (критично — это автономное поведение)
1. **OFF по умолчанию.** Самолечение включается флагом per-project (`self_heal: true` в topics.json) ИЛИ глобально env `SELF_HEAL_ENABLED`. Без флага — сканер работает как сейчас (только создаёт карточки, не чинит). НЕ включать ни для одного проекта по умолчанию.
2. **НИКОГДА не applied автоматически.** Агент-чинильщик доходит только до Review. Merge в основное дерево — ВСЕГДА руками Игоря (C2-apply). Это незыблемо.
3. **Лимит попыток.** Один инцидент → максимум 1 авто-починка. Если гейт после починки = risky → карточка в Failed с пометкой «авто-починка не прошла гейт», БОЛЬШЕ не пытаться (флаг `heal_attempted` в meta инцидента). Без этого — бесконечный цикл сжигания токенов.
4. **Лимит конкурентности.** Не запускать починку если проект занят (`running` lock) — как обычная карточка. Максимум N авто-починок одновременно по всем проектам (напр. 1-2) — не залить систему.
5. **Кто-чинит видно.** Каждое действие самолечения → Timeline (`kind:"self_heal"`) + пинг Игорю в TG. Полная наблюдаемость.
6. **Только git+clean проекты** (worktree-режим). Не-git/dirty → самолечение пропускает (нет безопасного отката).

## Что реализовать

### 1. Backend — флаг и условие
- Чтение флага: `_self_heal_enabled(project)` = `project.get("self_heal")` OR глобальный env. Default False.
- В `_scan_and_ingest` (или после него в loop): если включено И появились НОВЫЕ инцидент-карточки (added>0) И проект git+clean И не занят И инцидент не `heal_attempted` → запустить починку.

### 2. Backend — петля починки
- `_self_heal_card(ctx, project, incident_card)`:
  - Пометить инцидент `heal_attempted:true` (в meta/description) ДО запуска — чтобы при краше не зациклиться.
  - Сформировать промпт чинильщику: «На проекте инцидент: <title>. <описание ошибки/трейс из incident desc>. Найди причину и почини. Не трогай несвязанное. После — тесты должны проходить.»
  - Запустить через C2-механизм (worktree-режим, как перенос карточки в In Progress): `_card_worktree_setup` + `_run_card` с этим промптом. Переиспользовать существующий путь (НЕ дублировать SDK-цикл).
  - После прогона `_run_card` уже переносит в Review + пишет meta + git-diff (готово в C2).
  - [НОВОЕ] после починки автоматически вызвать гейт `_run_quality_gate(wt_path)` → записать вердикт в meta.
  - Если safe → карточка остаётся в Review, пометка «🔧 авто-починка · гейт ✓», пинг Игорю «готов фикс, проверь». Если risky → перенести в Failed, пометка «авто-починка не прошла гейт», пинг.
  - Все шаги → Timeline (`kind:"self_heal"`, статусы start/fixed/gate_ok/gate_fail).

### 3. Backend — интеграция в scanner loop
- В `_error_scanner_loop`: после `_scan_and_ingest`, если self-heal включён и есть новые чинибельные инциденты — поставить `asyncio.create_task(_self_heal_card(...))` (НЕ блокировать loop). С учётом лимита конкурентности (счётчик активных починок).

### 4. UI — индикация + переключатель
- В Overview проекта или настройках: тумблер «🔧 Самолечение» (вкл/выкл, пишет `self_heal` в topics через эндпоинт). Подпись: «Агент сам чинит новые ошибки в worktree и готовит фикс на твоё одобрение. Ничего не применяется без тебя.»
- Карточки авто-починки в Review/Failed визуально помечены (иконка 🔧, бейдж вердикта гейта).
- Эндпоинт `POST /api/projects/{id}/self-heal {enabled: bool}` — переключить флаг.
- `api.ts`, `types.ts`, `i18n/ru.ts`, ARIA.

### 5. Тесты (tests/test_self_healing.py)
- `_self_heal_enabled`: флаг проекта / env / default False.
- инцидент + heal включён → `_self_heal_card` запускается (мок run_engine + гейт); инцидент помечается heal_attempted ДО прогона.
- safe-гейт → карточка Review + пометка; risky → Failed + пометка.
- лимит: heal_attempted инцидент НЕ перезапускается.
- не-git проект → самолечение пропускается.
- занятый проект (running) → не запускается.
- self-heal OFF (default) → сканер только создаёт карточки, починку НЕ запускает (регрессия-страж: поведение без флага = как v0.7.0).
- Timeline получает self_heal-события.
- существующие 468 зелёные.

## Out of scope (НЕ в этой итерации — следующая магия)
- Авто-apply при safe (полная автономия) — НЕТ, человек всегда одобряет. Это сознательная граница.
- Внешние триггеры (Coolify webhook / CI fail / прод-ошибка → инцидент) — отдельный шаг 6. Сейчас триггер = существующий сканер (лог/тест).
- Мульти-агентная оркестрация (агент раздаёт агентам) — последней, отдельно.
- Самоулучшение (агент придумывает улучшения, не только чинит ошибки) — СЕВЕР, позже.

## Принцип
Минимум нового кода: переиспользуем сканер (триггер) + C2-worktree (изоляция) + гейт (проверка) + Timeline (наблюдаемость). Новое — только «клей»: условие запуска + оркестрация петли + предохранители. Граница автономии жёсткая: **агент готовит, человек применяет.** Это даёт 80% пользы «магии» при 0% риска неконтролируемых изменений в проде.

# Spec 006 — Память проекта (накапливаемая, в репо)

> Статус: APPROVED Игорем 2026-05-31. Реализация на ветке `feature-project-memory`.
> Карточка: новая (memory). Шаг 1 roadmap «полноценный сервис разработки».
> Рамка проекта: ИНСТРУМЕНТ для Игоря + OSS на будущее. БЕЗ привязки к ~/vault, Coolify, конкретной машине.

## Проблема
Агент каждую сессию — чистый лист. Не помнит: какие архитектурные решения приняты, на какие грабли наступали, что Игорь отверг и почему. Существующая «память» (`~/.claude/projects/<cwd>/memory/*.md`) — приватная (привязана к машине, НЕ в git проекта), read-only в кокпите, и агент САМ туда не пишет. Это блокер для всей будущей автономии (самолечащийся агент без памяти повторяет ошибки).

## Решение
Память проекта переезжает В РЕПО проекта: каталог **`.claude-ops/memory/`** в корне проекта, коммитится в git. Путешествует с проектом, видна при клоне (OSS), история — в git. Агент УМЕЕТ туда писать (а не только ты вручную). Кокпит позволяет читать/редактировать.

### Решение Игоря (зафиксировано)
- Память — **в репо проекта** (`.claude-ops/memory/`), не в ~/.claude. OSS-friendly.

## Структура `.claude-ops/memory/`
- `MEMORY.md` — индекс (одна строка на запись: `- [Заголовок](file.md) — хук`). Грузится агенту первым.
- `<slug>.md` — одна запись = один факт, с frontmatter:
  ```
  ---
  type: decision | gotcha | rejected | convention
  created: YYYY-MM-DD
  ---
  <суть. Для decision/rejected — **Почему:** причина.>
  ```
- Типы: `decision` (принятое решение + почему), `gotcha` (грабли, на которые наступили), `rejected` (что Игорь/проект отвергли + почему — чтобы не предлагать снова), `convention` (договорённость о стиле/подходе).

## Что реализовать

### 1. Backend — путь и хелперы
- `_project_memory_dir(cwd) -> Path` = `Path(cwd) / ".claude-ops" / "memory"`.
- ⚠️ Существующий `api_project_memory` читает из `_sdk_sessions_dir(cwd)/memory` (старое место). НОВОЕ поведение: читать из `.claude-ops/memory/`, НО для обратной совместимости — если нового нет, а старое есть, читать старое + показать баннер «миграция». (Миграцию старого→новое можно отдельной кнопкой/эндпоинтом; в этой итерации — просто читать новое место приоритетно.)
- Хелперы: `_memory_read_all(cwd)`, `_memory_write(cwd, name, content)`, `_memory_delete(cwd, name)`, `_memory_reindex(cwd)` (перестроить MEMORY.md из файлов). Все — anti-traversal (`_valid_*` имя файла, slug-паттерн), лимит размера (`_MEMORY_MAX_SIZE`).

### 2. Backend — агент умеет писать память
- Это ГЛАВНОЕ. Сейчас агент память не пишет. Варианты (выбрать в реализации проще/надёжнее):
  - (предпочтительно) Память в `.claude-ops/memory/` агенту доступна через обычные файловые инструменты (Read/Write) — он работает в cwd проекта, путь относительный, ему достаточно ЗНАТЬ про это место. → значит: добавить в шаблон `CLAUDE.md.tpl` раздел «Память проекта: пиши важные решения/грабли в `.claude-ops/memory/`», и в системную подсказку (operating-brief) короткую строку про это. Тогда агент сам ведёт память обычным Write — без новых API.
- НЕ раздувать operating-brief (gotcha проекта: тонкий nudge). Одна строка: «Важное решение/грабли → запиши в `.claude-ops/memory/` (см. CLAUDE.md)».

### 3. Endpoints (CRUD памяти из кокпита)
- `GET /api/projects/{id}/memory` — обновить под новый путь (`.claude-ops/memory/`), формат ответа сохранить ({files:[{name,content}], exists}).
- `POST /api/projects/{id}/memory/{name}` — создать/обновить запись (валидация slug, лимит). Обновляет MEMORY.md-индекс.
- `DELETE /api/projects/{id}/memory/{name}` — удалить запись + обновить индекс.
- Все под auth-middleware, anti-traversal, board-lock-стиль не нужен (файлы независимы), но atomic write.

### 4. Frontend — MemoryTab из read-only в редактируемый
- MemoryTab: список записей + просмотр (как сейчас) + **создать/редактировать/удалить** (как GlobalFilesTab умеет inline-edit — переиспользовать паттерн EditableMarkdown/Modal).
- Пустое состояние: обновить текст под новый путь `.claude-ops/memory/`.
- i18n-строки в ru.ts. ARIA на кнопках.

### 5. Шаблоны новых проектов
- `templates/CLAUDE.md.tpl` — добавить раздел «## Память проекта» с объяснением `.claude-ops/memory/` и когда писать (decision/gotcha/rejected/convention). Чтобы каждый новый проект сразу знал про память.
- `.gitignore.tpl` — НЕ игнорировать `.claude-ops/memory/` (память коммитится). Но `.claude-ops/` может содержать и приватное в будущем (ключи — spec 007) → игнорировать `.claude-ops/secrets/` заранее, оставить `.claude-ops/memory/` трекаемым.

## Тесты (tests/test_project_memory.py)
- `_project_memory_dir` возвращает правильный путь.
- write → read round-trip; MEMORY.md-индекс обновляется при write/delete.
- anti-traversal: имя `../x` → отказ; slug-валидация.
- лимит размера.
- API: GET пустой проект → exists:false; POST создаёт + индекс; DELETE убирает + индекс; bad name → 400.
- обратная совместимость: старое место читается если нового нет (если реализовано).
- существующие 325 тестов зелёные.

## Out of scope (этой итерации)
- Авто-суммаризация сессии в память (агент сам решает что важно по эвристике) — позже.
- Миграция-кнопка старое→новое (если сложно — просто читать новое приоритетно).
- Векторный поиск по памяти — не нужно, файлов мало, агент читает индекс.

## Принципы (OSS + инструмент)
- Ноль привязки к ~/vault / Coolify / машине Игоря. Память самодостаточна в репо.
- Память = обычный markdown, читается человеком и агентом одинаково.

# Spec 007 — Хранилище ключей проекта (.claude-ops/secrets/)

> Статус: APPROVED Игорем 2026-05-31. Ветка `feature-secrets-store`.
> Шаг 2 roadmap. Рамка: ИНСТРУМЕНТ Игоря + OSS. БЕЗ привязки к ~/vault, Coolify, машине.

## ⚠️ Граница (важно)
- **Vault Игоря (`~/vault/_system/Credentials.md`) НЕ трогаем, НЕ мигрируем, НЕ удаляем.** Он остаётся как личный источник истины Игоря.
- Мы строим ЧИСТЫЙ OSS-механизм хранилища секретов проекта с нуля. Для существующих проектов Игоря — он сам решит, что туда положить (вручную), мы ничего не переносим автоматически.

## Проблема
Сейчас агент не имеет безопасного способа получить секреты проекта (API-ключи, токены сервисов). В Vault — личное Игоря, не OSS и не изолировано по проектам. Для автономии (агент сам деплоит/дёргает API) нужно: секреты на проект, изолированно (ключи проекта А не видны Б), не в git, доступны агенту при выполнении.

## Решения Игоря (зафиксированы)
1. **Хранение:** plain `.env`-стиль файл `KEY=VALUE`, в `.claude-ops/secrets/`, gitignored, `chmod 600`. Шифрование — НЕ сейчас (можно добавить позже).
2. **Доступ агента:** через **env процесса** — при запуске задачи секреты проекта подмешиваются в env агента (`run_engine` уже принимает `env`). Агент видит как обычные переменные. Изоляция по cwd автоматическая.

## Файл секретов
- Путь: `<cwd>/.claude-ops/secrets/secrets.env`
- Формат: `KEY=VALUE` построчно (как `.env`), комментарии `#`, пустые строки игнор.
- `chmod 600` при создании/записи.
- В `.gitignore`-шаблоне `.claude-ops/secrets/` УЖЕ игнорируется (заложено в spec-006). Проверить, что и в реальном `.gitignore` проектов это есть (для не-наших — добавлять при первом сохранении секрета: дописать строку в `.gitignore` если её нет).
- ⚠️ Имена ключей: валидировать `^[A-Z_][A-Z0-9_]*$` (env-совместимые, anti-injection).

## Что реализовать

### 1. Backend — хелперы
- `_project_secrets_path(cwd) -> Path` = `<cwd>/.claude-ops/secrets/secrets.env`.
- `_secrets_read(cwd) -> dict[str,str]` — распарсить файл (KEY=VALUE). Если нет — {}.
- `_secrets_write(cwd, data: dict)` — atomic write (tmp+replace), chmod 600, создать dir. ⚠️ гарантировать .gitignore содержит `.claude-ops/secrets/` (дописать если нет).
- `_secrets_set(cwd, key, value)` / `_secrets_delete(cwd, key)` — точечно.
- Валидация ключа `^[A-Z_][A-Z0-9_]*$`. Лимит размера значения (напр. 8KB), числа ключей (напр. 100).

### 2. Backend — подача агенту через env (КЛЮЧЕВОЕ)
- В местах, где запускается `run_engine` с реальной задачей (НЕ только chat — также `_run_card`, `api_project_chat`), собрать env проекта из `_secrets_read(cwd)` и передать в `run_engine(env=...)`.
- ⚠️ Текущий `env` уже содержит `TG_CHAT_ID`/`TG_THREAD_ID` (bot.py:567) для TG. Секреты ДОПОЛНЯЮТ этот env, не затирают. Для web-чата/карточки env сейчас не задаётся — добавить.
- ⚠️ ИЗОЛЯЦИЯ: только секреты ЭТОГО cwd. Никогда не подмешивать секреты другого проекта (cwd — граница).
- ⚠️ Секреты НЕ должны попасть в audit-лог / сайдкары / транскрипт. Проверить: `audit()` пишет промпт+команды — убедиться что env туда не утекает. (env в audit не пишется сейчас — подтвердить.)

### 3. Endpoints (CRUD из кокпита)
- `GET /api/projects/{id}/secrets` — список **имён ключей** (НЕ значений! значения никогда не отдаём в API-ответ — только маска `***` или просто список ключей). `{keys: ["STRIPE_KEY", ...], exists: bool}`.
- `POST /api/projects/{id}/secrets/{key}` — задать значение (body {value}). Валидация имени, лимит. Ответ — обновлённый список ИМЁН (без значений).
- `DELETE /api/projects/{id}/secrets/{key}` — удалить ключ.
- Все под auth-middleware. ⚠️ Значения секретов НИКОГДА не возвращаются клиенту (даже авторизованному) — only-write, list-names. Это защита от утечки через браузер/логи.

### 4. Frontend — SecretsTab (или секция в Overview)
- Новая вкладка «🔑 Ключи» ИЛИ карточка в Overview. Решить проще: отдельный таб `secrets` (добавить в TabId).
- Показывает СПИСОК ИМЁН ключей (значения скрыты, показываем `STRIPE_KEY = ••••••`).
- Добавить ключ (имя + значение, значение в password-input), удалить ключ (ConfirmModal). Редактирование значения = повторный set (старое значение в UI не показываем).
- Подсказка: «Ключи доступны агенту как переменные окружения при выполнении задач в этом проекте. Не коммитятся в git.»
- `api.ts`: `secrets(id)`, `setSecret(id,key,value)`, `deleteSecret(id,key)`. `types.ts`, `i18n/ru.ts`, ARIA.

### 5. Шаблоны
- `templates/CLAUDE.md.tpl`: раздел «## Секреты проекта» — объяснить `.claude-ops/secrets/secrets.env`, что доступны как env-переменные, не коммитятся, задаются через кокпит (вкладка Ключи) или вручную.

## Безопасность (критично — это ключи!)
- Значения НИКОГДА не возвращаются клиенту через API (list-names-only).
- Секреты НЕ в git (gitignore), НЕ в audit/сайдкар/транскрипт.
- chmod 600 на файле.
- Имена ключей строго валидируются (env-injection).
- Изоляция по cwd — жёстко.
- anti-traversal на {key} в путях эндпоинтов.

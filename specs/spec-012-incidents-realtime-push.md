## Зачем (три большие идеи)

1. **Ошибки → карточки почти мгновенно, без «призраков».** Сейчас (spec-011 Ф1) фоновый сканер раз в 5 мин перечитывает `journalctl -n 300` целиком → (а) задержка до 5 мин, (б) перечитывает СТАРЫЕ строки в хвосте окна и воскрешает уже закрытые инциденты (реально словили 2026-06-04: удалил карточку — сканер пересоздал её из строки 07:43, которая ещё в окне). Лечится **high-water-mark**: помнить позицию последней обработанной строки и читать только НОВОЕ.
2. **Логи — универсальный движок (это и есть OSS-преимущество), push — опциональный апгрейд.** Любой сервис/бот/CLI на любом языке уже пишет логи → zero-coupling: «дай команду логов — получи канбан инцидентов». Push (проект сам зовёт `POST /incident`) НЕ требовать, а предлагать тем, кому нужно мгновенно/структурно. Свои ошибки кокпита идут быстрым in-process путём (без круга через лог).
3. **Самолечение — осторожный слой, отвязанный от источника.** Больше real-time + больше источников = больше триггеров heal. Без дедупа/дебаунса/рейт-лимита это шторм агент-ранов (тот flood = 124 события → было бы 124 worktree-запуска). Поэтому safety-слой делаем ДО открытия push-источников.

## Решения (ПРИНЯТЫ 2026-06-04)

1. **Кнопка «🩺 Скан» в шапке — УБРАТЬ.** Сейчас она тихая (результат выбрасывается, подтверждения нет) и маргинальная. После Ф0 (инкрементальный near-real-time скан) ручной триггер не нужен. Если когда-нибудь понадобится «проверить сейчас» — это место на вкладке **Доска** рядом с инцидентами, с явным фидбэком (тост «просканировано N, +M»), а не молчаливая кнопка в шапке.
2. **Кнопка «Прогнать тесты» — ОСТАВИТЬ, но с настоящим выводом.** Запуск тестов по требованию полезен. Бэкенд (`api_project_test`) уже отдаёт `{detected, ok, cmd, exit_code, timed_out, output}` — показать инлайн сводку «✓ прошли (pytest, exit 0) / ✗ упали (exit 1)», а клик по результату → модалка с полным `output` (какие тесты упали). Хватит выбрасывать `output` в галочку.
3. **Логи-first, push opt-in.** Никогда не делать push обязательным условием мониторинга проекта. Проект без интеграции должен мониториться по логам как есть.
4. **Самолечение — OFF по умолчанию остаётся незыблемым** (наследие spec-010). Новый триггер «появился инцидент» НЕ снимает ни один предохранитель spec-010 (clean-tree worktree, только до Review, heal_attempted один раз, лимит конкурентности).

## ФАЗА 0 — Инкрементальный скан (high-water-mark) + чистка кнопок

**Backend — сканер только по НОВЫМ строкам лога:**
- Ввести per-project **курсор позиции** в `data/` (напр. `data/scan_state.json`: `{<project_id>: {cursor|since|offset, last_scan_ts}}`). Хелперы `_scan_state_load/_scan_state_save` (глотают ошибки, как Timeline).
- В `_scan_and_ingest` / `_run_log_cmd` для сканера: читать лог инкрементально по типу источника:
  - **journald** (`journalctl -u …`): использовать `--after-cursor <cursor>` + `--show-cursor`, сохранять новый курсор. Это нативный high-water-mark journald.
  - **docker** (`docker logs …`): `--since <last_scan_ts>`.
  - **файл/прочее** (`tail`/`cat`): byte-offset, при усечении (offset > размер) — сбросить в 0.
  - Детект типа — по началу команды (`journalctl` / `docker` / иначе file/generic). Неизвестный тип → fallback на текущее поведение (последние N строк), но с дедупом по hash против существующих карточек (как сейчас), чтобы не словить призраков.
- **Дедуп остаётся** (`_ingest_errors_to_board` по hash) — high-water-mark это слой СВЕРХУ, не замена: если карточка существует, бампим seen/last; новую не плодим.
- **Закрытый инцидент не воскресает:** курсор продвигается за обработанные строки → старая строка повторно не читается. Доп. страховка от «удалил карточку → пересоздалась»: вести `data/dismissed_incidents.json` (`hash → dismissed_ts`), сканер НЕ создаёт карточку для hash, закрытого < `_DISMISS_TTL` (напр. 24ч) назад. DELETE/move-to-done err-карточки → записывает hash в dismissed.
- **Снизить интервал** скана до near-real-time (напр. `ERROR_SCAN_INTERVAL` default 30–60с — дёшево, т.к. читаем только новое). Параметр оставить в глобальных настройках.

**Frontend — кнопки (решения 1–2):**
- `ProjectView.tsx`: удалить `HeaderScanBtn` и его использование. `api.scanErrors` оставить (бэкстоп/Доска), но из шапки убрать.
- `HeaderTestRunner`: показывать сводку (`ok` + `cmd` + `exit_code`/`timed_out`), клик по результату → модалка (`Modal.tsx`) с полным `output` в `<pre>`. Не сбрасывать результат по таймеру в 6с — пусть висит до следующего запуска/закрытия.

**Acceptance:**
- [ ] Удалил err-карточку — на следующем скане она НЕ возвращается (пока ошибка не повторится свежей строкой).
- [ ] Новая ошибка в логе → карточка в Failed в пределах интервала (≤60с), не 5 мин.
- [ ] Скан не перечитывает строки старше курсора (проверяемо тестом на journald `--after-cursor`).
- [ ] «Скан» в шапке отсутствует; «Тесты» показывают вывод (модалка), а не только символ.

## ФАЗА 1 — Свои ошибки кокпита → карточка in-process (мгновенно)

- `error_middleware` ловит необработанные исключения САМОГО кокпита (claude-ops-bot). Сейчас только логирует `UNHANDLED …` → ждёт сканер. Сделать **прямой путь**: middleware (кроме benign-disconnect, см. gotcha) вызывает `_ingest_errors_to_board(HERE, "claude-ops-bot", [err])` → карточка появляется мгновенно, без круга через лог и без зависимости от high-water-mark.
- Лог `UNHANDLED …` ПИСАТЬ всё равно (аудит-трейл + бэкстоп). Дедуп по hash гарантирует, что сканер не задвоит карточку, которую middleware уже создал.
- Вынести «создать инцидент-карточку из одного err-dict» в переиспользуемый хелпер (`_report_incident(ctx, project, err)`), чтобы им пользовались и middleware (Ф1), и push-эндпоинт (Ф3).

**Acceptance:**
- [ ] Необработанное исключение в хендлере кокпита → карточка в Failed мгновенно (не через 5 мин/60с).
- [ ] benign-disconnect (ConnectionReset) по-прежнему НЕ создаёт карточку (gotcha spec-011).
- [ ] Сканер не задваивает уже созданную middleware карточку.

## ФАЗА 2 — Самолечение: дебаунс / дедуп / рейт-лимит (safety ДО push)

Триггер heal перевести с «сканер нашёл added>0» на **«появился новый инцидент-инстанс»** (из любого источника), но обвешать предохранителями, чтобы real-time/push не штормили:
- **Дебаунс по инциденту:** не запускать heal, пока ошибка не «устоялась» — напр. heal только если инцидент существует > `_HEAL_DEBOUNCE` (напр. 60–120с) ИЛИ `seen >= N` (повторилась). Транзиентный единичный блип карточку создаёт, но heal не дёргает.
- **Глобальный рейт-лимит heal:** не больше `_HEAL_MAX_PER_WINDOW` запусков за окно (напр. 3/час на проект). Превышение → лог + пропуск (не плодить worktree-шторм).
- **benign/ignore-лист:** классы, которые НИКОГДА не лечим (ConnectionReset, CancelledError, TimeoutError на стримах…). Конфиг в настройках проекта.
- Все предохранители spec-010 СОХРАНЯЮТСЯ (OFF-default, clean-tree worktree, heal_attempted один раз, только до Review, лимит конкурентности). Этот слой — СВЕРХУ.
- Наблюдаемость: Timeline `kind:"self_heal"` фиксирует и причину ПРОПУСКА (debounce/rate-limit/benign), не только запуск.

**Acceptance:**
- [ ] Симуляция flood (одна ошибка ×100 за минуту) → ≤ rate-limit запусков heal, не 100.
- [ ] Единичный транзиентный инцидент (seen=1, молодой) → карточка есть, heal НЕ запущен.
- [ ] benign-класс из ignore-листа → карточка (опц.) есть, heal никогда.

## ФАЗА 3 — Опциональный push-эндпоинт + проводка в шаблонах (OSS upgrade)

- **`POST /api/projects/{id}/incident`** (или generic `/api/incident` с полем project) — принимает `{exc_class, where/path, excerpt, ts?}`, дёргает `_report_incident` (Ф1) → карточка + (через Ф2) возможный heal. **Авторизация:** проектный токен (переиспользовать механику секретов проекта `.claude-ops/secrets/` — выдать `CLAUDEOPS_INCIDENT_TOKEN`), не общий cookie. Рейт-лимит на эндпоинт.
- **Дедуп общий** с лог-путём (тот же hash) — push и лог-сканер одной ошибки = одна карточка.
- **Шаблоны error-handler (`templates/CLAUDE.md.tpl`, секция `## Error Handler`):** добавить ОПЦИОНАЛЬНЫЙ сниппет fire-and-forget push на кокпит (как эталон networking-os CRM `services/alerts.py` → `asyncio.create_task(alert_exception(...))`, но целью — `/incident`). Подчеркнуть: **опционально**, базовый мониторинг работает и без него (по логам).
- `## ClaudeOps conformance`: добавить строку `- incident push: нет` (само-декларация, не обязательна).

**Acceptance:**
- [ ] `POST /incident` с валидным токеном → карточка мгновенно; без токена → 401/403.
- [ ] Один и тот же инцидент по push И в логе → одна карточка (дедуп).
- [ ] Проект без push по-прежнему мониторится по логам (push нигде не обязателен).

## Инварианты (не нарушать)
- **Логи — это zero-coupling база; push никогда не обязателен.** Любой проект мониторится, отдав только `log_cmd`.
- **Дедуп по hash — единая точка** (`_ingest_errors_to_board`): лог-сканер, in-process middleware, push — все через неё; одна ошибка = одна карточка.
- **Закрытый инцидент не воскресает** (high-water-mark + dismissed-TTL).
- **Самолечение OFF по умолчанию + все предохранители spec-010** + новый слой (debounce/rate-limit/benign). Никогда не auto-apply, только до Review.
- **benign-disconnect (ConnectionReset) — не инцидент** (gotcha spec-011, сохранить во всех путях).

## Что НЕ делать (scope)
- Не делать push обязательным / не выпиливать лог-сканер (он бэкстоп для не-push проектов и для крашей, умирающих раньше, чем успеют пушить — systemd OnFailure).
- Не трогать предохранители spec-010 (только добавлять слой сверху).
- Не строить очередь/брокер (Kafka и пр.) — JSONL/файл-состояние достаточно (стиль Timeline).
- Не превращать «Тесты» в фоновый прогон (после spec-011 тесты — только по кнопке).
- Не добавлять модель/прочие кнопки в шапку (решение spec-011).
- **Не трогать кнопки «Аудит проекта» / «Апгрейд»** в Обзоре (`ProjectStructureCardFull`) — это quality-функции (агент аудитит/доводит проект против baseline → карточки в Backlog), вне scope монитора инцидентов. Оставить как есть; максимум — уточнить подпись/фидбэк отдельно.

## Как проверить (после деплоя)
1. Уронить тестовый эндпоинт кокпита (ValueError) → карточка в Failed мгновенно (Ф1).
2. Удалить err-карточку → подождать интервал → НЕ вернулась (Ф0 dismissed/cursor).
3. Сгенерить серию одинаковых ошибок → одна карточка, heal ≤ rate-limit (Ф2).
4. `curl POST /incident` с токеном проекта → карточка; без токена → 403 (Ф3).
5. Шапка: «Скан» нет; «Тесты» → модалка с выводом упавших тестов.

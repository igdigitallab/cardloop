# Spec 008 — Timeline (наблюдаемость: единая лента событий проекта)

> Статус: APPROVED (автономно по рамке Игоря). Ветка `feature-timeline`.
> Шаг 3 roadmap. Рамка: ИНСТРУМЕНТ Игоря + OSS. БЕЗ привязки к Coolify/Vault/машине.

## Проблема
События проекта (что агент делал, прогоны карточек, инструменты, ошибки) сейчас **эфемерны**: `_bus_publish` транслирует подписчикам в реальном времени и они исчезают. Есть `audit`-лог (плоский текст `data/audit/*.log`), но он не структурирован, не привязан к карточкам/сессиям и не виден в кокпите. Для доверия к автономии нужно: **видеть всю историю** — что агент сделал, когда, в какой задаче, с каким исходом. Без наблюдаемости автономный агент = чёрный ящик.

## Решение
Дать шине **персистентность**: каждое событие из `_bus_publish` дополнительно пишется в JSONL-лог проекта. Кокпит получает вкладку «Timeline» — хронологическая лента (история + live через существующий SSE). Это надстройка над уже работающей шиной, минимально инвазивная.

## Дизайн-решения (приняты автономно по рамке)
1. **Хранение:** JSONL `data/timeline/<project-slug>.jsonl` (append-only, в `data/` — НЕ в git, как audit/sessions; data/ уже gitignored). Не в репо проекта (это операционный лог рантайма, не часть проекта; в отличие от памяти, которая — знание проекта).
   - OSS-обоснование: data/ локальна каждой инсталляции, не путешествует. Это правильно для логов.
2. **Точка записи:** единая — внутри `_bus_publish` (туда уже стекаются ВСЕ события run_start/tool/text/run_end + enriched session_key). Один хук = ноль дублирования.
3. **Ротация:** лимит размера файла (напр. 5 МБ) → ротация в `<slug>.jsonl.1` (одна), чтобы не пухло бесконечно. Чтение Timeline берёт текущий + .1.
4. **Связь с карточками:** события уже несут `run_id` (= card_id) и `source` (card/chat/tg). Timeline группирует/фильтрует по этому.

## Формат события (JSONL, одна строка = одно событие)
```json
{"ts": 1730000000.123, "session_key": "chat:thread", "kind": "run_start|tool|text|run_end", "source": "card|chat|tg", "run_id": "<card_id|->", ...поля_события}
```
- `ts` добавляется при записи (time.time()). Остальное — как уже формирует `_bus_publish`.
- `text`-события: обрезать до разумного (напр. 2000 симв) чтобы лог не пух от длинных ответов.
- ⚠️ Секреты НЕ должны попасть (они в env, не в событиях — но проверить tool-bash события: команда может содержать `$KEY`-подстановку? нет — agent видит env, а в команде обычно `$VAR`; на всякий случай НЕ логировать env). Не логируем поля env.

## Что реализовать

### 1. Backend — персистентность шины
- `_timeline_path(project_slug_or_cwd) -> Path` = `DATA/timeline/<slug>.jsonl`. Slug из cwd (как `_sdk_sessions_dir` делает замену `/`→`-`, или basename — выбрать стабильный).
- `_timeline_append(session_key, event)` — добавить `ts`, сериализовать, append. Обрезка `text`. Ротация по размеру (5MB→.1). Ошибки глотать (не ломать прогон — как `_bus_publish` drop-on-full).
- Вызвать `_timeline_append` ВНУТРИ `_bus_publish` (единая точка). ⚠️ session_key→project: маппинг через ctx topics (session_key содержит chat:thread; найти проект/cwd по нему — есть `_find_project_by_*`/topics). Если не резолвится — писать в `_unknown.jsonl` или по session_key. Не падать.
- ⚠️ `_bus_publish` синхронная — запись файла синхронная append (быстро, ок) ИЛИ через очередь. Проще: синхронный append с try/except. Не блокировать надолго.

### 2. Backend — эндпоинты чтения
- `GET /api/projects/{id}/timeline?limit=N&before=ts` — вернуть последние N событий (по умолчанию 200), с пагинацией по ts (для подгрузки истории). Читает `<slug>.jsonl` (+`.1` если нужно), парсит, отдаёт массив (новые первыми или хронологически — решить, для ленты обычно новые внизу).
- Опц.: `GET /api/projects/{id}/timeline/stats` — сводка (сколько прогонов, ошибок, последняя активность). Если просто — добавить, иначе skip.
- Live-поток уже есть (`/api/projects/{id}/activity-stream`) — Timeline-вкладка подписывается на него для новых событий, не нужен новый SSE.

### 3. Frontend — TimelineTab
- Новый таб `timeline` («📜 Timeline» или «🕒 Лента»): добавить в TabId + ProjectView.
- Загрузка истории через `GET /timeline` + live-дополнение через подписку на activity-stream (паттерн useProjectActivity уже есть — переиспользовать).
- Рендер ленты: иконка по kind/source (🔧 tool, 💬 text, ▶️ run_start, ✅/❌ run_end), время, краткое описание (для tool — _format_tool уже даёт kind/cmd/file). Группировка по run_id (прогон карточки = свёрнутый блок) — если просто; иначе плоская лента с метками.
- Фильтр по source (карточка/чат/TG) и/или по kind — минимально (хотя бы «только ошибки/прогоны»). Если раздувает — плоская лента + позже фильтры.
- Подгрузка истории (кнопка «раньше» / infinite scroll по before=ts).
- `api.ts`: `timeline(id, opts)`. `types.ts`: `TimelineEvent`. `i18n/ru.ts`. ARIA. CSS.

### 4. Тесты (tests/test_timeline.py)
- `_timeline_path` стабильный slug.
- `_timeline_append` пишет JSONL, добавляет ts, обрезает text, ротация при превышении.
- `_bus_publish` теперь пишет в timeline (интеграция: publish → файл содержит событие).
- эндпоинт GET возвращает события, пагинация before/limit.
- секреты/env НЕ попадают в timeline (тест).
- битый jsonl-лайн не валит чтение (graceful).
- существующие 421 зелёные.

## Безопасность / краевые
- env/секреты НЕ в timeline.
- Ротация — чтобы не пух диск.
- Запись не ломает прогон (try/except, как drop-on-full в шине).
- Чтение graceful к битым строкам.
- anti-traversal на project resolve (как везде).

## Out of scope (позже)
- Единый Timeline по ВСЕМ проектам (глобальная лента) — пока per-project.
- Внешние события (Coolify/CI/прод-ошибки) в Timeline — это шаг 5 (триггеры), не сейчас.
- Поиск/экспорт timeline.
- Метрики/графики.

## Принцип
Timeline = надстройка над уже работающей шиной `_bus_publish`. Минимум инвазии: одна точка записи, переиспользуем существующий SSE и форматтер `_format_tool`. Это фундамент доверия для автономии (шаг 5).

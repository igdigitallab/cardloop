# Spec 009 — Авто-гейт качества (тесты/линт перед «Применить»)

> Статус: APPROVED (автономно по рамке Игоря). Ветка `feature-quality-gate`.
> Шаг 4 roadmap. Рамка: ИНСТРУМЕНТ Игоря + OSS. БЕЗ привязки к Coolify/Vault.

## Проблема
C2-«Применить» (`api_card_apply`) делает `git merge --no-ff` ветки карточки в основное дерево **на глаз** — пользователь смотрит diff и жмёт ✓. Но он не знает, **не сломал ли агент тесты/линт**. Для доверия (и тем более для будущей автономии) нужно: перед применением сам прогнать тесты/линт проекта и сказать **«безопасно / рискованно»**.

## Решение
Перед merge в `api_card_apply` (опционально, по флагу/настройке) прогоняется автодетект-тест проекта **в worktree карточки** (там изменения агента, основное дерево чистое). Результат → вердикт `safe | risky | unknown`. UI показывает вердикт ПЕРЕД подтверждением применения. Применение не блокируется жёстко (пользователь решает), но видит риск.

## Дизайн-решения (приняты автономно по рамке)
1. **Где гонять:** в **worktree карточки** (`meta.wt_path`), НЕ в основном дереве. Там лежат изменения агента до merge — это и есть «проверить перед вливанием». Основное дерево не трогается.
2. **Что гонять:** переиспользовать `_detect_test_cmd(cwd)` (уже есть: pytest-конфиг/tests/, npm test, make test). Линт — если есть `npm run lint` или `ruff`/`flake8` конфиг (опц., добавить детект; если сложно — только тесты в этой итерации, линт отметить out-of-scope).
3. **Когда:** отдельный эндпоинт `POST .../check` (прогнать гейт и вернуть вердикт), вызывается из UI ДО apply. НЕ встраивать жёстко в apply (apply остаётся быстрым; гейт — осознанный шаг). Опц.: apply может принять `?checked=true`.
4. **Вердикт:**
   - `safe` — тесты прошли (rc=0).
   - `risky` — тесты упали (rc≠0) или таймаут.
   - `unknown` — не нашёл как тестировать (нет конфигов) → не блокируем, просто инфо «тестов нет».
5. **Не блокируем жёстко.** Пользователь видит вердикт и сам решает применять. (Жёсткая блокировка — позже, для автономного контура, не для ручного режима.)

## Что реализовать

### 1. Backend — гейт-хелпер
- `_run_quality_gate(wt_path) -> dict` — прогнать `_detect_test_cmd(wt_path)` В worktree, вернуть `{verdict: "safe|risky|unknown", tests: {detected, ok, cmd, exit_code, output(обрезан), timed_out}, lint: {...}|None}`.
  - Переиспользовать логику `api_project_test` (вынести общий `_run_tests_in(cwd)` если красиво, или вызвать детект+subprocess как там). Таймаут 300с, обрезка вывода.
  - Линт (опц.): если `web/package.json` со скриптом lint ИЛИ ruff/flake8 — прогнать. Если усложняет — пропустить, отметить.
- ⚠️ Гонять в `meta.wt_path` (worktree карточки), проверить что путь существует (404-safe).
- Секреты проекта: подмешать в env прогона (как делает run для карточки) — тесты могут требовать ключи. Через `_secrets_read(cwd)` основного проекта.

### 2. Endpoint
- `POST /api/projects/{id}/tasks/{card}/check` — прочитать meta (worktree-режим, как apply), прогнать `_run_quality_gate(wt_path)`, вернуть вердикт. Legacy/нет worktree → `{verdict:"unknown", reason:"legacy"}`. Auth, `_valid_card_id`, anti-traversal.
- Записать результат гейта в meta-сайдкар (`_write_run_meta`: `gate: {verdict, ts}`) — чтобы UI помнил и Timeline видел.
- (Опц.) публиковать событие гейта в `_bus_publish` → попадёт в Timeline («гейт: safe/risky»). Хорошо для наблюдаемости.

### 3. Frontend
- В модалке результата карточки (там же где ✓ Применить / ✗ Отмена в Review, worktree-режим): добавить кнопку **«🧪 Проверить»** (запускает check) + показ вердикта:
  - 🟢 Безопасно (тесты прошли) / 🔴 Рискованно (тесты упали, показать вывод) / ⚪ Тестов нет.
  - После check кнопка «Применить» получает визуальный акцент по вердикту (зелёная при safe, предупреждение при risky — но НЕ блокирована).
  - Вывод тестов — сворачиваемый блок (как сайдкар).
- `api.ts`: `checkCard(id, card)`. `types.ts`: `GateResult`. `i18n/ru.ts`. ARIA.

### 4. Тесты (tests/test_quality_gate.py)
- `_run_quality_gate`: проект с проходящими тестами → safe; с падающими → risky; без тестов → unknown.
- гонит в worktree (не в основном дереве) — проверить cwd прогона.
- секреты подмешиваются в env гейта.
- API check: worktree-карточка → вердикт; legacy → unknown; bad card_id → 400; нет worktree → 400/404-safe.
- meta обновляется полем gate.
- (если линт реализован) lint-вердикт.
- существующие 453 зелёные.

### 5. Доки
- `docs/API.md` (+check). `ARCHITECTURE.md` (гейт в потоке C2: run → review → **check** → apply/discard). `CLAUDE.md` (gotcha: гейт гонит тесты в worktree, не блокирует). `CHANGELOG.md` (Unreleased → v0.7.0).

## Файлы

- `web/src/tabs/BoardTab.tsx` (модалка результата — кнопка Проверить + вердикт), `api.ts`, `types.ts`, `i18n/ru.ts`, CSS.
- `tests/test_quality_gate.py`.
- Docs.

## Безопасность / краевые
- Гейт гонит ПРОИЗВОЛЬНЫЙ тест-код проекта (это и так делает api_project_test) — таймаут 300с, kill при превышении, обрезка вывода.
- Секреты в env гейта — но НЕ в выводе/Timeline (вывод тестов может содержать — обрезаем, но не маскируем; отметить: тесты не должны печатать секреты. Для OSS — это ответственность тестов проекта).
- Worktree-путь валидируется (существует).
- anti-traversal на card_id.

## Out of scope (позже)
- Жёсткая блокировка apply при risky (для автономного контура — шаг 5).
- Кастомные гейт-команды (сейчас автодетект).
- Покрытие/метрики качества.
- Линт — если усложняет, в этой итерации только тесты.

## Принцип
Гейт = переиспользование `_detect_test_cmd` + прогон в worktree (где уже изолированы изменения C2). Минимум нового кода. Это превращает C2-«Применить» из «нажми и молись» в осознанное решение с данными. Фундамент для автономного самолечения (шаг 5): там вердикт станет автоматическим условием.
